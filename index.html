Day 1: Breaking Things to Secure Them (My Cryptography Lab)

Lab Log Cryptography

Day one of any cybersecurity journey is usually spent fighting installation wizards[cite: 619]. For me, that meant setting up Oracle’s VirtualBox and booting up a pre-built Kali Linux image[cite: 620]. I quickly learned that Kali is a rolling-release operating system, meaning it constantly updates its packages to ensure its heavy-hitting security tools never suffer from a lack of optimization[cite: 621].

Once my sandbox environment was rock-solid, I dove straight into the core of security: Cryptography—the art of transforming plain, readable text into scrambled, unreadable ciphertext[cite: 622]. The ultimate goal here is simple: protecting confidentiality and proving authenticity[cite: 623].

Things got real when I began splitting cryptography into its two primary workflows: Symmetric and Asymmetric[cite: 624].

1. The Key Dilemma: Symmetric vs. Asymmetric

Symmetric encryption uses a single, shared private key to both lock and unlock data[cite: 625]. It’s incredibly lightweight and fast, but it suffers from a massive roadblock known as the key distribution problem[cite: 626]. If you have to share that secret key over a public network to let someone decrypt your message, you run an immense risk of interception[cite: 627].

Asymmetric encryption completely solves this by deploying a mathematically linked key pair: a Public Key (which you give to the world to encrypt data) and a Private Key (which only you keep secret to decrypt it)[cite: 628].

To test this, I used OpenSSL (a built-in Linux crypto library) to play with AES-256 encryption[cite: 629]. It breaks data into 128-bit blocks and scrambles them using a key length that yields 2256 total combinations[cite: 630]. To put that absolute unit of a wall into perspective: if all the supercomputers on Earth worked simultaneously, it would still take over 3 × 1051 years (millions of billions of trillions of years) to guess the right key via brute force[cite: 631].

Yet, I still managed to break it on my first try... sort of[cite: 632].

Checkpoint: Troubleshooting OpenSSL

I ran straight into a classic bad magic number error[cite: 633]. Because I accidentally inverted my input flags, I fed a plaintext file into a decryption command[cite: 634]. OpenSSL checked the very beginning of the file for its mandatory file signature bytes (Salted__), failed to find them, and stopped immediately[cite: 635]. Once I corrected the file flags, my message decrypted flawlessly[cite: 636].

2. GPG Keys & One-Way Hashes

Next, I moved on to GPG (GNU Privacy Guard) to set up an asymmetric identity[cite: 637]. I generated my key pair using modern Elliptic Curve Cryptography (cv25519), exported my public key, and simulated a recipient-only data transfer[cite: 638].

Finally, I experimented with Hashing using SHA-256[cite: 639]. Unlike encryption, hashing is a strictly one-way, deterministic process that acts as a digital fingerprint for data integrity[cite: 640]. I even simulated a tampering attack by modifying just a single character in my text file[cite: 641]. Thanks to the avalanche effect, the final 64-character hexadecimal string transformed entirely—proving how easy it is to catch a malicious actor altering data[cite: 642].

Checkpoint: Digital Signature Verification

I wrapped up the lab by using my private key to clear-sign a document, establishing non-repudiation and proving that the document explicitly came from me[cite: 643]. First lab is officially in the books, and my engineering blog is live on GitHub Pages[cite: 644]!
